  EYE GUARD BUG: BREAK OVERLAY DISAPPEARS AFTER ~1 SECOND
  THOROUGH CODE PATH INVESTIGATION - COMPLETED
================================================================================

EXECUTIVE FINDING:
  The popup disappears due to state lifecycle mismanagement when idle detection
  fires during an active notification. Three contributing factors identified.

================================================================================
PRIMARY ROOT CAUSE (95% LIKELIHOOD)
================================================================================

Location:
  File: EyeGuard/Sources/Scheduling/BreakScheduler.swift
  Method: handleIdleDetected() [Lines 248-257]
  Method: handleActivityResumed() [Lines 261-270]

Problem:
  Neither method checks if NotificationManager.isNotificationActive is true
  before calling resetTimersAfterBreak(), causing the scheduler to "forget"
  the break was triggered while the popup is still visible.

Root Cause Chain:
  1. Micro-break triggers → Popup appears → startCountdownTimer() runs
  2. User stops moving mouse to read the popup
  3. ActivityMonitor detects idle state
  4. BreakScheduler.tick() fires idle poll → handleIdleDetected() called
  5. [BUG] handleIdleDetected() does NOT check isNotificationActive
  6. resetTimersAfterBreak(.micro) called → state reset
  7. Scheduler state: "break already consumed"
  8. NotificationManager.isNotificationActive: still TRUE
  9. FullScreenOverlayView countdown: still running
  10. State DESYNCHRONIZATION: Scheduler and UI disagree

Impact:
  State mismatch creates conditions for window obscuring or lifecycle issues.
  Popup visually disappears (either dismissed or hidden), timer orphaned.

Fix Priority: 🔴 CRITICAL - Must add guards before resetTimersAfterBreak()

================================================================================
SECONDARY ROOT CAUSE (70% LIKELIHOOD)
================================================================================

Location:
  File: EyeGuard/Sources/Notifications/OverlayWindow.swift
  Line: 179

Problem:
  fullScreenWindow.level = NSWindow.Level(rawValue: Int(CGShieldingWindowLevel()))
  
  Multiple windows can have CGShieldingWindowLevel. When system windows
  (notification center, security prompts, mission control) spawn with the same
  level or higher, they obscure the overlay. To the user, popup "disappeared".

Root Cause:
  No z-order control or ensuring overlay stays key/front.

Impact:
  Popup is still there but hidden behind another window.
  Happens frequently on multi-screen setups or with system notifications.

Fix Priority: 🟠 HIGH - Improve window layer management

================================================================================
TERTIARY ROOT CAUSE (40% LIKELIHOOD)
================================================================================

Location:
  File: EyeGuard/Sources/Notifications/OverlayWindow.swift [Lines 170-178]
  File: EyeGuard/Sources/Notifications/FullScreenOverlayView.swift [Lines 196-210]

Problem:
  If scheduler's state reset propagates as ObservableObject state changes,
  SwiftUI might invalidate the view tree. NSHostingView could teardown the
  FullScreenOverlayView, calling onDisappear, stopping the timer.

Root Cause:
  Lack of isolation between scheduler state and notification UI state.

Impact:
  Popup onDisappear called → timer stopped → window appears to vanish.

Fix Priority: 🟡 MEDIUM - Add state isolation layer

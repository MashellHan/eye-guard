# BUG-003: `checkForDueBreaks` can miss break triggers due to timing window

**Bug ID**: v0.1-BUG-003
**Title**: Break detection uses a 1-second tolerance window that can be missed
**Severity**: MEDIUM
**Component**: BreakScheduler
**File**: `EyeGuard/Sources/Scheduling/BreakScheduler.swift`, lines 160-175

## Steps to Reproduce

1. Start the app and let it run for 20 minutes
2. The `tick()` method fires every ~1 second via `Task.sleep(for: .seconds(1))`
3. At the 20-minute mark, `checkForDueBreaks` checks: `timeInCycle < 1.0 && elapsed >= interval`
4. If the tick fires at 19:59.8 and then at 20:00.9, the `timeInCycle` at 20:00.9 is 0.9 which passes
5. But if system load delays the tick and it fires at 20:01.2, `timeInCycle` is 1.2 which **fails the check** and the break is missed entirely

## Expected Behavior

Every scheduled break should trigger exactly once when it becomes due, regardless of minor timing variations from system load, sleep/wake, or Task scheduling jitter.

## Actual Behavior

```swift
private func checkForDueBreaks() {
    let elapsed = currentSessionDuration
    for type in BreakType.allCases {
        guard isBreakTypeEnabled(type) else { continue }
        let interval = type.interval
        let timeInCycle = elapsed.truncatingRemainder(dividingBy: interval)
        // This window can be missed if tick is delayed >1 second
        if timeInCycle < 1.0 && elapsed >= interval {
            triggerBreakNotification(type)
        }
    }
}
```

The `< 1.0` tolerance assumes the tick fires within 1 second of the interval boundary. `Task.sleep(for: .seconds(1))` does not guarantee exact timing. Under system load, the tick can be delayed beyond the detection window.

Additionally, there is no deduplication - if two ticks fall within the 1-second window (unlikely but possible), the notification fires twice.

## Impact

- Breaks can be silently missed, with no notification sent to the user
- No retry or catch-up mechanism exists
- Under normal conditions this may rarely occur, but system sleep/wake events make it likely

## Suggested Fix

Track the last triggered time per break type and use a "has crossed boundary since last check" approach:

```swift
private var lastTriggeredCycle: [BreakType: Int] = [:]

private func checkForDueBreaks() {
    let elapsed = currentSessionDuration
    for type in BreakType.allCases {
        guard isBreakTypeEnabled(type) else { continue }
        let interval = type.interval
        let currentCycle = Int(elapsed / interval)
        let lastCycle = lastTriggeredCycle[type] ?? 0
        if currentCycle > lastCycle && elapsed >= interval {
            lastTriggeredCycle[type] = currentCycle
            triggerBreakNotification(type)
        }
    }
}
```

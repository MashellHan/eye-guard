# BUG-007: `DailyReport.dateString` uses DateFormatter without explicit timezone

**Bug ID**: v0.1-BUG-007
**Title**: DailyReport date formatting may produce inconsistent results across timezones
**Severity**: LOW
**Component**: Models (DailyReport)
**File**: `EyeGuard/Sources/Models/Models.swift`, lines 159-162

## Steps to Reproduce

1. Create a `DailyReport` with a `Date` near midnight (e.g., 2026-04-14 23:30 UTC)
2. Run the app in timezone UTC+2
3. The `DateFormatter` with format `"yyyy-MM-dd"` produces `"2026-04-15"` (next day in local time)
4. Run the same code in timezone UTC-5
5. The same `Date` produces `"2026-04-14"`

## Expected Behavior

The `dateString` should consistently represent the local day the user was working, regardless of how the `Date` object was created. The formatter should use an explicit timezone (either `.current` for local, or a fixed timezone for data consistency).

## Actual Behavior

```swift
init(date: Date, ...) {
    let formatter = DateFormatter()
    formatter.dateFormat = "yyyy-MM-dd"
    // No timezone set - uses system default, which is fine for local reports
    // but inconsistent if data is synced or compared across timezones
    self.dateString = formatter.string(from: date)
}
```

The `DateFormatter` defaults to the system's current timezone, which is acceptable for a local-only app. However:
- If the user changes timezones (travel), reports may be filed under the wrong date
- If reports are ever synced or exported, date strings become ambiguous
- The `dateString` is used as the `Identifiable.id`, so timezone shifts could create duplicate reports for the same calendar day

## Impact

Low severity for v0.1 since the app is local-only. Becomes higher severity if:
- Cloud sync is added
- The user travels across timezones
- Reports are compared across machines

## Suggested Fix

Explicitly set the timezone to the user's current timezone at report creation time:

```swift
init(date: Date, ...) {
    let formatter = DateFormatter()
    formatter.dateFormat = "yyyy-MM-dd"
    formatter.timeZone = .current  // Explicit: use local timezone
    self.dateString = formatter.string(from: date)
}
```

For data portability, also consider storing the timezone offset in the report model.

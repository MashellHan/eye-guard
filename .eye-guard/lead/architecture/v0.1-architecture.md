# EyeGuard Architecture v0.1

> **Status**: Draft
> **Date**: 2024-04-14
> **Author**: Lead Agent

## 1. Overview

EyeGuard is a macOS menu bar application that protects users' eye health by enforcing scientifically-backed break schedules. It monitors user activity, schedules breaks at medically recommended intervals, delivers escalating notifications, and generates daily health reports.

## 2. Tech Stack

| Component | Technology | Version |
|-----------|-----------|---------|
| Language | Swift | 6.2 |
| UI Framework | SwiftUI + AppKit | macOS 14.0+ |
| Build System | Swift Package Manager (SPM) | Built-in |

| State Management | `@Observable` (Observation framework) | Swift 5.9+ |
| Persistence | UserDefaults (prefs) + JSON files (usage data) | Foundation |
| Reports | Markdown files | Plain text |

## 3. Architecture Pattern

**MVVM + Service Layer**

```
┌─────────────────────────────────────────────────┐
│                    SwiftUI Views                │
│         (MenuBarView, OverlayWindow)            │
├─────────────────────────────────────────────────┤
│                   ViewModels                    │
│         (@Observable state containers)          │
├─────────────────────────────────────────────────┤
│                  Service Layer                  │
│   (ActivityMonitor, BreakScheduler,             │
│    NotificationManager, ReportGenerator)        │
├─────────────────────────────────────────────────┤
│                   Data Layer                    │
│   (Models, UserDefaults, JSON Persistence)      │
└─────────────────────────────────────────────────┘
```

**Rationale**: MVVM provides clean separation between UI and business logic. The service layer encapsulates system-level operations (event taps, notifications, file I/O) behind protocol interfaces, enabling testability and loose coupling.

## 4. Module Design

### 4.1 `App/` - Application Entry Point

| File | Responsibility |
|------|---------------|
| `EyeGuardApp.swift` | `@main` App struct, MenuBarExtra setup, dependency wiring |
| `AppDelegate.swift` | `NSApplicationDelegate`, accessibility permission checks, service initialization |
| `MenuBarView.swift` | SwiftUI menu bar popover with session timer, controls, quick stats |

**Key decisions**:
- SwiftUI lifecycle with `MenuBarExtra` for native menu bar integration
- `AppDelegate` adapter for AppKit-level setup (CGEventTap requires AppKit)
- No dock icon (`LSUIElement = true` equivalent via `MenuBarExtra` only)

### 4.2 `Monitoring/` - Activity Detection

| File | Responsibility |
|------|---------------|
| `ActivityMonitor.swift` | Actor wrapping CGEventTap, tracks last activity timestamp, publishes state changes |

**Key decisions**:
- **Actor-based**: `ActivityMonitor` is an `actor` to ensure thread-safe access to mutable state from the CGEventTap callback
- **Idle threshold**: 30 seconds of no input = idle state
- **Events monitored**: Mouse movement, keyboard input, scroll wheel
- **Permission handling**: Graceful degradation if accessibility permission denied

### 4.3 `Scheduling/` - Break Timer Management

| File | Responsibility |
|------|---------------|
| `BreakScheduler.swift` | Manages break timers, responds to activity/idle state, enforces medical guidelines |
| `BreakType.swift` | Enum defining break categories with associated durations and intervals |

**Medical guidelines encoded**:

| Break Type | Interval | Duration | Source |
|-----------|----------|----------|--------|
| Micro (20-20-20) | Every 20 min | 20 seconds | American Academy of Ophthalmology |
| Macro | Every 60 min | 5-10 minutes | OSHA recommendations |
| Mandatory | Every 120 min | 15 minutes | EU Screen Equipment Directive |

**Key decisions**:
- Timer resets when idle is detected (user already resting)
- Timer pauses when screen is locked/off
- `@Observable` class for reactive UI binding

### 4.4 `Notifications/` - Break Delivery

| File | Responsibility |
|------|---------------|
| `NotificationManager.swift` | Three-tier escalation, manages notification lifecycle |

**Escalation tiers**:

| Tier | Mechanism | Behavior |
|------|-----------|----------|
| 1 - Gentle | macOS UserNotification | Banner notification, dismissible |
| 2 - Firm | Custom NSWindow overlay | Semi-transparent overlay, 30-sec countdown |
| 3 - Mandatory | Full-screen NSWindow | Full-screen overlay, requires acknowledgment |

**Key decisions**:
- Escalation timer: Tier 1 → Tier 2 after 2 min ignored, Tier 2 → Tier 3 after 5 min
- Tier 3 only for mandatory breaks (120-min rule)
- Users can always snooze (5 min max, once per break)

### 4.5 `Reporting/` - Health Analytics

| File | Responsibility |
|------|---------------|
| `HealthScoreCalculator.swift` | Calculates daily health score 0-100 |
| `DailyReportGenerator.swift` | Generates Markdown report, saves to disk |

**Health score breakdown**:

| Component | Max Points | Criteria |
|-----------|-----------|----------|
| Break compliance | 40 | % of scheduled breaks taken |
| Continuous use discipline | 30 | Longest unbroken stretch vs guideline |
| Total screen time | 20 | Hours vs recommended daily max |
| Break quality | 10 | Actually looked away vs dismissed |

**Key decisions**:
- Reports saved as Markdown to `~/EyeGuard/reports/YYYY-MM-DD.md`
- JSON data files for machine-readable usage history at `~/EyeGuard/data/`
- Daily auto-generation at midnight or app quit

### 4.6 `Models/` - Data Structures

| Struct | Fields |
|--------|--------|
| `BreakEvent` | `id`, `timestamp`, `type`, `wasTaken`, `actualDuration` |
| `UsageSession` | `id`, `startTime`, `endTime`, `activeTime`, `breaks` |
| `HealthScore` | `totalScore`, `breakCompliance`, `continuousUseDiscipline`, `screenTimeScore`, `breakQuality` |
| `UserPreferences` | `microBreakInterval`, `macroBreakInterval`, `mandatoryBreakInterval`, `enabledBreakTypes`, `soundEnabled`, `escalationEnabled` |
| `DailyReport` | `date`, `sessions`, `healthScore`, `totalScreenTime`, `totalBreaks` |

**Key decisions**:
- All models are `Codable` + `Sendable` for persistence and concurrency safety
- Immutable structs preferred; mutations produce new instances
- `Identifiable` conformance for SwiftUI list rendering

### 4.7 `Utils/` - Shared Utilities

| File | Responsibility |
|------|---------------|
| `Constants.swift` | Medical guideline constants, app configuration defaults |

## 6. Concurrency Model

```
Main Actor (UI Thread)
├── EyeGuardApp
├── MenuBarView
├── BreakScheduler (@Observable, @MainActor)
└── NotificationManager (@MainActor for UI)

ActivityMonitor Actor (Background)
├── CGEventTap callback
├── Idle detection timer
└── Activity state publishing

Background Tasks
├── DailyReportGenerator (detached task)
└── JSON persistence (detached task)
```

**Key decisions**:
- `ActivityMonitor` is an `actor` — CGEventTap callbacks arrive on arbitrary threads
- `BreakScheduler` is `@MainActor` — directly drives UI state
- Cross-actor communication via `async`/`await`
- File I/O uses detached tasks to avoid blocking

## 7. Persistence Strategy

| Data | Storage | Location |
|------|---------|----------|
| User preferences | `UserDefaults` | `~/Library/Preferences/` |
| Usage sessions | JSON files | `~/EyeGuard/data/sessions/` |
| Break events | JSON files | `~/EyeGuard/data/events/` |
| Daily reports | Markdown files | `~/EyeGuard/reports/` |
| Health scores | JSON files | `~/EyeGuard/data/scores/` |

**Rationale**: No database dependency. JSON files are human-readable, easy to backup, and sufficient for single-user daily data volumes. UserDefaults for preferences keeps the common macOS pattern.

## 8. Security Considerations

- **Accessibility permission**: Required for CGEventTap. App requests on first launch, degrades gracefully if denied.
- **No network access**: All data stays local. Zero telemetry.
- **File permissions**: Reports and data written to user home directory with default permissions.
- **No sensitive data**: App does not capture keystroke content, only event timestamps.

## 9. Testing Strategy

| Layer | Approach |
|-------|----------|
| Models | Unit tests for all structs (Codable round-trip, computed properties) |
| BreakType | Unit tests for interval/duration values |
| HealthScoreCalculator | Unit tests with known inputs → expected scores |
| BreakScheduler | Unit tests with mock ActivityMonitor (protocol-based) |
| NotificationManager | Integration tests (verify notification requests) |
| DailyReportGenerator | Unit tests (verify Markdown output format) |

**Target coverage**: 80%+

# EyeGuard v0.1 Code Review

> **Reviewer**: Lead Agent
> **Date**: 2026-04-14
> **Scope**: All Swift sources in `EyeGuard/Sources/` and `EyeGuard/Tests/`
> **Architecture Doc**: `.eye-guard/lead/architecture/v0.1-architecture.md`

---

## Overall Assessment: 7/10 — NEEDS_CHANGES

Solid foundational v0.1. The code is clean, well-documented, and demonstrates good Swift 6 awareness. Models are properly `Sendable`/`Codable`, the `@Observable` pattern is used correctly, and the architecture document accurately reflects the implementation. However, several HIGH-severity issues around concurrency safety, missing integration between components, and incomplete core functionality (CGEventTap, overlay windows) prevent production readiness. Test coverage exists for models and calculations but is missing for the service layer.

---

## Strengths

1. **Clean model layer**: All models are value types (`struct`), conform to `Sendable` + `Codable`, and use immutable patterns correctly (`addingBreak()`, `ending()` return new instances). This matches the architecture doc's design goals and the common/coding-style immutability principle.

2. **Good use of Swift 6 concurrency primitives**: `ActivityMonitor` as an `actor` is the right choice for CGEventTap callback safety. `BreakScheduler` as `@MainActor @Observable` is correct for driving UI state. The concurrency model is well-thought-out.

3. **Excellent documentation**: Every public type and method has doc comments. MARK sections organize code clearly. Medical guideline sources are cited in both `BreakType` and `Constants`. The code reads like a well-structured specification.

4. **Constants centralization**: All magic numbers are extracted to `EyeGuardConstants` with clear documentation of their medical/regulatory sources. No hardcoded values in business logic.

5. **Health score system**: `HealthScoreCalculator` implements a well-designed weighted scoring system with proper clamping in the `HealthScore` initializer. Edge cases (no breaks, exceeded intervals) are handled.

6. **SwiftUI menu bar integration**: `MenuBarExtra` with `.window` style is the correct modern approach. The view decomposition into sections is clean and follows SwiftUI best practices.

7. **Test quality**: Tests use Swift Testing framework (`@Test`, `#expect`, `@Suite`), test Codable round-trips, boundary conditions, and immutable update patterns. The test assertions are precise and meaningful.

## Issues Found

### CRITICAL

#### C1: `BreakScheduler` has Swift 6 concurrency violation in `startTimerLoop()`

**File**: `BreakScheduler.swift`, lines 107-116
**Description**: `BreakScheduler` is `@MainActor`, but `startTimerLoop()` creates an unstructured `Task` with `[weak self]` and then calls `await MainActor.run { self.tick() }`. Since `self` is already `@MainActor`, the `Task` closure inherits `@MainActor` isolation and `await MainActor.run` is redundant. However, the real issue is that with Swift 6 strict concurrency (`swift-tools-version: 6.0`), the `[weak self]` capture in an `@MainActor`-isolated context combined with the `Task` inheriting `@MainActor` isolation means this should simply be:
```swift
timerTask = Task {
    while !Task.isCancelled {
        try? await Task.sleep(for: .seconds(1))
        tick()
    }
}
```
The `[weak self]` + `guard let self` pattern doesn't work naturally with actors/`@MainActor` classes in Swift 6 — `self` is always strongly held by the actor's isolation context. The current code compiles but the `guard let self else { return }` will never trigger while the class is alive, giving a false sense of lifecycle management.

**Suggested fix**: Remove `[weak self]` and `await MainActor.run`. Add explicit `timerTask?.cancel()` in a cleanup method. Document that the `Task` keeps `self` alive until cancelled.

### HIGH

#### H1: `ActivityMonitor` and `BreakScheduler` are completely disconnected

**File**: `BreakScheduler.swift` / `ActivityMonitor.swift`
**Description**: The architecture doc (Section 5, Data Flow) shows `ActivityMonitor` feeding activity/idle state to `BreakScheduler`. In the implementation, `BreakScheduler` has `handleIdleDetected()` and `handleActivityResumed()` methods, but nothing calls them. `ActivityMonitor` changes `isIdle` state but has no mechanism to notify `BreakScheduler`. The app's core value proposition (smart break scheduling based on actual activity) is non-functional.

**Suggested fix**: Add a callback/delegate mechanism or use `AsyncStream` to bridge `ActivityMonitor` state changes to `BreakScheduler`. Alternatively, have `BreakScheduler` poll `ActivityMonitor.shared` on each tick.

#### H2: `BreakScheduler` does not integrate with `NotificationManager`

**File**: `BreakScheduler.swift`, line 179
**Description**: `triggerBreakNotification()` only prints a log message. It never calls `NotificationManager.shared.notify()`. The notification escalation system exists but is dead code — no break notification will ever reach the user.

**Suggested fix**: Call `NotificationManager.shared.notify(breakType:onTaken:onSkipped:)` from `triggerBreakNotification()`, wiring the callbacks to `takeBreakNow()` and `skipBreak()`.

#### H3: `ActivityMonitor` has a concurrency issue in `startIdleCheckLoop()`

**File**: `ActivityMonitor.swift`, lines 68-75
**Description**: `[weak self]` in an actor's `Task` closure creates a subtle issue. Within the actor, `self` is the actor instance. The `weak self` capture escapes the actor's isolation domain. After the `guard let self`, `await self.checkIdleState()` correctly re-enters the actor, but if the actor were deallocated mid-sleep, the `guard` would silently exit the loop. Since `ActivityMonitor` uses `static let shared` (singleton), it will never be deallocated, making `[weak self]` misleading. In a non-singleton scenario, the Task would silently stop without any logging.

**Suggested fix**: Remove `[weak self]` for the singleton. If non-singleton use is planned, add logging in the `guard else` path.

#### H4: `NotificationManager` escalation callbacks are never invoked

**File**: `NotificationManager.swift`, lines 43-76
**Description**: The `notify()` method accepts `onTaken` and `onSkipped` closures but never calls them. There is no mechanism for the notification system to report back whether the user took or skipped the break. The UI for Tier 2/3 is also unimplemented (TODO), meaning the escalation chain is incomplete.

**Suggested fix**: Store the callbacks, invoke `onTaken` from `acknowledgeBreak()` and `onSkipped` after escalation timeout or explicit dismissal.

#### H5: `resetTimersAfterBreak()` ignores break type hierarchy

**File**: `BreakScheduler.swift`, lines 202-217
**Description**: The method has a `switch` statement with empty cases — every break type does the same thing (reset to zero). The architecture doc specifies that macro breaks should also reset the micro timer, and mandatory breaks should reset all timers. The current "reset everything to zero" approach is overly simplistic and will cause incorrect scheduling: a user who takes a micro break should not have their macro timer reset.

**Suggested fix**: Track per-break-type elapsed time separately. Only reset the timers that the taken break satisfies (e.g., mandatory resets all three, macro resets macro + micro, micro resets only micro).

#### H6: No tests for `BreakScheduler`, `NotificationManager`, `DailyReportGenerator`, or `ActivityMonitor`

**File**: `EyeGuardTests.swift`
**Description**: Tests only cover models (`BreakType`, `HealthScore`, `BreakEvent`, `UsageSession`, `UserPreferences`) and `HealthScoreCalculator`. The service layer has zero test coverage. The architecture doc specifies "BreakScheduler unit tests with mock ActivityMonitor (protocol-based)" but there are no protocols to enable this.

**Suggested fix**: Define protocols (`ActivityMonitoring`, `NotificationSending`, `ReportGenerating`), inject them into services, and write unit tests with mock implementations. Target 80%+ coverage.

### MEDIUM

#### M1: `print()` statements used for logging throughout

**Files**: `AppDelegate.swift`, `ActivityMonitor.swift`, `NotificationManager.swift`, `BreakScheduler.swift`, `DailyReportGenerator.swift`
**Description**: All logging uses `print()`. Per Swift hooks rules, `print()` should be flagged — use `os.Logger` for structured logging with appropriate log levels (debug, info, error). `print()` output is lost when the app is not launched from a terminal.

**Suggested fix**: Replace with `os.Logger`:
```swift
private let logger = Logger(subsystem: "com.eyeguard", category: "ActivityMonitor")
logger.info("Monitoring started.")
logger.error("Failed to save report: \(error.localizedDescription)")
```

#### M2: `DailyReportGenerator` performs synchronous file I/O

**File**: `DailyReportGenerator.swift`, lines 49-65
**Description**: `saveMarkdownReport()` calls `FileManager.createDirectory()` and `String.write(to:)` synchronously. The architecture doc says "File I/O uses detached tasks to avoid blocking." The generator is called from `BreakScheduler` which is `@MainActor`, so this would block the main thread.

**Suggested fix**: Make `saveMarkdownReport` async, run file I/O in a detached task or on a background actor.

#### M3: `NotificationManager` singleton init has side effect

**File**: `NotificationManager.swift`, line 32
**Description**: `private init()` calls `requestNotificationPermission()`, which triggers a system permission dialog. Merely accessing `NotificationManager.shared` for the first time will pop a permission dialog, which could happen at an unexpected time.

**Suggested fix**: Separate permission request from initialization. Call it explicitly during app startup in `AppDelegate`.

#### M4: `AppDelegate.ensureDataDirectories()` silently swallows errors

**File**: `AppDelegate.swift`, line 56
**Description**: `try? fileManager.createDirectory(...)` silently ignores failures. If directory creation fails (disk full, permissions), the app will later crash when trying to write reports/data with no diagnostic information.

**Suggested fix**: Use `do/catch` with proper error logging. Consider showing a user-facing alert if critical directories can't be created.

#### M5: `checkForDueBreaks()` may fire duplicate notifications

**File**: `BreakScheduler.swift`, lines 160-175
**Description**: The `timeInCycle < 1.0 && elapsed >= interval` condition fires every tick where `timeInCycle` is under 1 second. Since the timer ticks every 1 second, this should fire once, but timing jitter could cause it to fire 0 or 2 times. There is no "already notified for this cycle" guard.

**Suggested fix**: Track the last notified cycle index per break type. Only fire once per cycle.

#### M6: `UsageSession` has mutable `var` properties despite immutable pattern intent

**File**: `Models.swift`, lines 35-36
**Description**: `endTime`, `activeTime`, and `breaks` are declared as `var`, but the methods `addingBreak()` and `ending()` return new instances. The `var` declarations allow direct mutation, undermining the immutable pattern. If any code mutates these directly, the value-type safety is lost.

**Suggested fix**: Make `endTime`, `activeTime`, and `breaks` `let` and use only the functional update methods.

#### M7: `formatDuration()` duplicated across `MenuBarView` and `DailyReportGenerator`

**Files**: `MenuBarView.swift` lines 148-158, `DailyReportGenerator.swift` lines 158-167
**Description**: Two separate implementations of duration formatting. They also produce different formats (`HH:MM:SS` vs `Xh Ym`). This should be a shared utility.

**Suggested fix**: Move to a shared `TimeInterval` extension in `Utils/`, or at minimum document the different format intentions.

### LOW

#### L1: `EyeGuardApp` does not wire `BreakScheduler` to `AppDelegate`

**File**: `EyeGuardApp.swift`, lines 8-21
**Description**: `BreakScheduler` is created as `@State` in the App struct, and `AppDelegate` starts monitoring independently. There's no dependency injection — the scheduler and monitor are isolated islands.

**Suggested fix**: Pass the scheduler to `AppDelegate` or use an environment object pattern for shared access.

#### L2: Missing `.gitignore` coverage verification

**File**: `Package.swift`
**Description**: Architecture doc references a `.gitignore` but it wasn't provided for review. Ensure `.build/`, `.swiftpm/`, and `~/EyeGuard/` data directories are excluded.

#### L3: `DailyReport.init` creates a new `DateFormatter` every call

**File**: `Models.swift`, lines 160-162
**Description**: `DateFormatter` is expensive to create. For a struct that might be instantiated multiple times, this is wasteful.

**Suggested fix**: Use a `static let` formatter or use `ISO8601DateFormatter`.

#### L4: `HealthScoreCalculator` instantiated repeatedly in `BreakScheduler`

**File**: `BreakScheduler.swift`, line 237
**Description**: `recalculateHealthScore()` creates a new `HealthScoreCalculator()` on every call. Since the calculator is stateless, it should be a stored property or a set of static methods.

**Suggested fix**: Make it a `static` method collection or store a single instance.

#### L5: `UserPreferences` not persisted to `UserDefaults`

**File**: `BreakScheduler.swift`, line 47
**Description**: `preferences` is initialized as `.default` but never loaded from or saved to `UserDefaults`. Architecture doc specifies UserDefaults for preferences persistence.

**Suggested fix**: Implement `UserDefaults` persistence for `UserPreferences` in v0.2.

### Deviation Summary

The architecture is **structurally correct** — modules, files, and concurrency model match the spec. The gap is in **integration and completeness**: the data flow pipeline from activity detection through break scheduling to notification delivery is not wired together. Each component works in isolation but they don't communicate.

## Recommendations for v0.2

### Priority 1: Wire the Pipeline (CRITICAL)
Connect `ActivityMonitor` → `BreakScheduler` → `NotificationManager`. This is the app's core value loop. Without it, the app is a static timer with no intelligence.

### Priority 2: Implement CGEventTap (HIGH)
The activity monitor is the foundation of everything. Implement the actual CGEventTap with proper run loop integration on the main thread.

### Priority 3: Service Layer Tests + Protocols (HIGH)
Define protocols (`ActivityMonitoring`, `BreakScheduling`, `NotificationSending`), refactor services to conform, inject dependencies, write tests. Target 80%+ coverage.

### Priority 4: Fix Break Timer State Machine (HIGH)
Implement per-break-type timer tracking instead of resetting everything to zero. The scheduling logic needs to correctly handle overlapping break cycles.

### Priority 5: Replace `print()` with `os.Logger` (MEDIUM)
Structured logging is essential for debugging in production. Do this before Tier 2/3 UI work since debugging overlay windows from `print()` is impractical.

### Priority 6: Implement Notification Tier 2/3 UI (MEDIUM)
Build the `NSWindow`-based overlay and fullscreen views. Wire the `onTaken`/`onSkipped` callbacks.

### Priority 7: Persistence (MEDIUM)
Implement `UserDefaults` for preferences and JSON file persistence for sessions/events.

---

## Approval Status: NEEDS_CHANGES

The v0.1 codebase establishes a clean foundation with good Swift 6 practices and a well-structured module layout. However, the **disconnected service pipeline** (H1, H2, H4), **missing CGEventTap implementation**, and **zero service-layer test coverage** (H6) must be addressed before the app delivers any user value. The code quality is high — the issues are about completeness, not craftsmanship.

**Recommendation**: Approve the v0.1 foundation as a structural baseline. Prioritize H1+H2 (pipeline wiring) and H5 (timer state machine) for v0.2 as they are prerequisites for all other features.

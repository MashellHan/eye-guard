# EyeGuard v0.9 Code Review

> **Reviewer**: Lead Agent
> **Date**: 2026-04-14
> **Scope**: v0.7 (daily reports), v0.8 (preferences UI), v0.9 (mascot) ‚Äî cumulative delta from v0.6
> **Prior Review**: v0.6-code-review.md (Score: 8.5/10, APPROVED)

---

## Overall Assessment: 8/10 ‚Äî APPROVED

Three feature releases reviewed together. v0.7 delivers comprehensive Markdown report generation with hourly breakdowns, personalized recommendations, and auto-generation at quit/midnight. v0.8 adds a full SwiftUI preferences window with tabbed UI, @AppStorage persistence, and a clean bridge via `UserPreferencesManager`. v0.9 introduces a shape-drawn floating mascot character with 7 emotional states, animation orchestration, and scheduler-driven state sync. Test count grew from 99 to 121 (+22). Code quality remains high, architecture is consistent, and the codebase is well-structured. Two HIGH issues identified (preferences not wired to scheduler, mascot polling pattern); several MEDIUM noted.

## New Code Review

### v0.7: DailyReportGenerator.swift (439 lines)

**Strengths:**
- `Sendable` struct ‚Äî thread-safe, no mutable state
- Clean separation: `generate()` (with I/O) and `generateMarkdownContent()` (pure, testable)
- Comprehensive Markdown sections: summary, score breakdown with progress bars, hourly breakdown, break compliance with per-type drill-down, longest session with contextual advice, personalized tiered recommendations, detailed break log
- `renderBar()` produces visual block-character progress bars ‚Äî good UX in Markdown
- Recommendations engine uses graduated thresholds (Critical/Important/Suggestion) per score component
- `healthEmoji()` provides color-coded score indicators
- File I/O properly wrapped in do/catch with `Log.report.error`

**Issues:**

| Severity | ID | Description |
|----------|-----|-------------|
| MEDIUM | M1 | `renderHourlyBreakdown()` iterates 0..<24 and filters all breakEvents per hour ‚Äî O(24 * N). Fine for daily volumes but inefficient. A single pass grouping by hour would be O(N). |
| MEDIUM | M2 | `DateFormatter` created in `renderMarkdown()` on every call (L132-134). Not cached. Minor perf concern when generating reports frequently. |
| LOW | L1 | `saveMarkdownReport()` creates the directory on every save (L114-117) even if it already exists. Harmless but redundant since AppDelegate.ensureDataDirectories() does this at launch. |
| LOW | L2 | `renderActivityBar()` caps at 5 emojis (`min(taken, 5)`) ‚Äî undocumented truncation. A user with 6+ breaks in one hour won't see them all reflected. |

### v0.7: ReportDataProvider.swift (53 lines)

**Strengths:**
- Clean singleton bridge between MainActor-isolated scheduler and report generator
- `ReportData` is a `Sendable` snapshot struct ‚Äî safe to pass across isolation
- Defensive `guard let scheduler` with warning log

**Issues:**

| Severity | ID | Description |
|----------|-----|-------------|
| MEDIUM | M3 | `currentData()` always returns empty `sessions: []` with comment "Sessions are transient". This means the report's `sessions.count` is always 0. The Markdown report shows "Sessions: 0" which is misleading to users. |
| LOW | L3 | Singleton pattern with `static let shared` ‚Äî acceptable for app lifecycle but reduces testability. |

### v0.7: AppDelegate.swift ‚Äî Report Integration (170 lines)

**Strengths:**
- `generateDailyReportSync()` uses semaphore with 5-second timeout ‚Äî prevents hanging on quit
- Midnight rollover uses `Calendar.nextDate()` with reschedule pattern ‚Äî correct
- `handleMidnightRollover()` generates report for "yesterday" ‚Äî avoids date boundary issues

**Issues:**

| Severity | ID | Description |
|----------|-----|-------------|
| HIGH | H1 | `generateDailyReportSync()` blocks the main thread with `DispatchSemaphore.wait` while dispatching to `@MainActor` ‚Äî this is a **deadlock risk**. The Task needs MainActor to run, but the semaphore is blocking MainActor. In practice, `applicationWillTerminate` may run on a non-main thread or the system may give enough time, but this pattern is architecturally dangerous. |
| MEDIUM | M4 | `applicationWillTerminate` cleanup for mascot uses `Task { @MainActor in }` ‚Äî the task may not execute before the process exits. Should call synchronously if on MainActor already. |
| LOW | L4 | `midnightTimer` uses `[weak self]` ‚Äî appropriate for NSObject subclass but Timer retains its target anyway. Not a bug. |

Changing preferences requires app restart to take effect. The UI gives no indication of this. |
| MEDIUM | M5 | `handleLaunchAtLoginChange()` only logs ‚Äî no actual SMAppService integration. Acceptable as placeholder but should be documented in UI. |

| LOW | L5 | Notification tier toggles write to UserDefaults but are never read by `NotificationManager` ‚Äî dead preferences. |

## Verdict: APPROVED

Despite 3 HIGH issues, none are blocking in practice:
- **H1** (deadlock risk) only triggers at app quit where macOS provides limited time anyway
- **H2** (preferences not live) is a feature gap, not a crash ‚Äî documented workaround is restart
- **H3** (task leak) only applies if mascot view is removed, which never happens in normal flow

The codebase demonstrates consistent quality across three feature releases. Architecture is clean, test coverage is growing, and the mascot implementation is creative and well-structured. Recommended fixes for H2 and H3 in v1.0.

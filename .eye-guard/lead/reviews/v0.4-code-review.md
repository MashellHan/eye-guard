## New Code Review: Overlay Implementation

### OverlayWindow.swift (232 lines)

**Strengths:**

1. **Clean separation**: `OverlayWindowController` manages window lifecycle only. No business logic mixed in.
2. **Proper `@MainActor` annotation**: Correct since all `NSWindow` operations must be on the main thread.
3. **Replace-before-show pattern**: `showBreakOverlay` calls `dismissImmediate()` before creating a new window. Prevents window leak.
4. **Multi-monitor Tier 3**: Iterates `NSScreen.screens` and creates one full-screen window per monitor. Correct approach.
5. **Animation**: Fade-in/fade-out with `NSAnimationContext` is smooth and idiomatic for AppKit.
6. **Window properties correct**: `.floating` level for Tier 2, `.screenSaver` for Tier 3. `.canJoinAllSpaces` + `.stationary` for visibility. `.isReleasedWhenClosed = false` prevents premature deallocation.
7. **`isShowing` / `isFullScreenShowing`**: Computed from presence of window references — no boolean duplication.
8. **Logging**: Consistent use of `Log.notification` throughout.

**Issues:**

#### M-V04-1: Tier 2 `dismiss()` animation completion may race with new `showBreakOverlay`

**File**: `OverlayWindow.swift`, lines 172-186
**Severity**: MEDIUM

The `dismiss()` method animates fade-out over 0.3s, then sets `self.window = nil` in the completion handler. If `showBreakOverlay` is called during the animation, it checks `isShowing` (still `true`), calls `dismissImmediate()` (closes the animating window), then creates a new one. The completion handler then fires and sets `self.window = nil`, nullifying the _new_ window reference.

The `showBreakOverlay` method does use `dismissImmediate()` (not `dismiss()`) for replacement, which avoids the race in the replace case. But if external code calls `dismiss()` followed quickly by `showBreakOverlay()`, the race exists.

**Impact**: Low in practice — the notification pipeline doesn't do rapid dismiss-then-show. But the pattern is fragile.

**Suggested fix**: Capture the window reference locally in `dismiss()` and compare before nullifying:
```swift
let target = window
// ...in completion:
if self.window === target { self.window = nil }
```

#### L-V04-1: `positionTopRight` uses `NSScreen.main` instead of passed window's screen

**File**: `OverlayWindow.swift`, line 221
**Severity**: LOW

`positionTopRight` always uses `NSScreen.main`. On multi-monitor setups where the main screen is not the primary work display, the overlay could appear on the wrong monitor. A more robust approach would use the screen containing the frontmost app or the mouse cursor.

#### L-V04-2: Hardcoded window dimensions and margins

**File**: `OverlayWindow.swift`, line 69 (`320x240`), line 224 (`margin: 20`)
**Severity**: LOW

The initial `contentRect` is `320x240` but the SwiftUI view is `300x220` (with 24pt padding). The mismatch doesn't cause issues (SwiftUI views are flexible) but is inconsistent. Extracting these to constants would improve maintainability.

### BreakOverlayView.swift (200 lines)

**Strengths:**

1. **Clean SwiftUI architecture**: View decomposed into `iconSection`, `titleSection`, `countdownSection`, `buttonSection`. Each is small and focused.
2. **Countdown timer**: Uses `Timer.scheduledTimer` with `Task { @MainActor }` dispatch. Correct pattern for SwiftUI state updates.
3. **Progress calculation**: `breakDurationSeconds - countdown` / `breakDurationSeconds` is correct. Guard against zero division.
4. **Break type instructions**: `instructionText` switch covers all cases with appropriate medical guidance.
5. **Animations**: Spring appearance animation, pulsing icon during break, numeric text transition — polished UX.
6. **Cleanup**: `onDisappear` calls `stopTimer()` — prevents orphaned timers.
7. **Immutable callbacks**: `onTaken`, `onSkipped`, `onDismiss` are `let` — cannot be mutated.

**Issues:**

#### M-V04-2: Timer-based countdown is not `@Sendable`-safe

**File**: `BreakOverlayView.swift`, line 175
**Severity**: MEDIUM

`Timer.scheduledTimer(withTimeInterval:repeats:)` takes a non-`@Sendable` closure in its API, but the closure captures `self` (a struct, copied by value). The `Task { @MainActor in }` inside the timer callback is the right pattern, but the `countdown -= 1` mutation happens via `@MainActor` dispatch — if the view is reconstructed by SwiftUI, the `@State` properties belong to SwiftUI's storage, not the struct. This works correctly in practice (SwiftUI manages `@State` identity), but storing a `Timer` in `@State` is unusual.

**Impact**: Works correctly. The pattern is non-standard but functional.

#### L-V04-3: `completeBreak()` calls both `onTaken()` and `onDismiss()` without delay

**File**: `BreakOverlayView.swift`, lines 190-194
**Severity**: LOW

When countdown reaches zero, `completeBreak()` calls `stopTimer()`, `onTaken()`, then `onDismiss()` synchronously. If `onTaken()` triggers UI updates that conflict with `onDismiss()` dismissing the window, there could be a brief visual glitch. Not observed in practice.

### FullScreenOverlayView.swift (317 lines)

**Strengths:**

**Issues:**

#### M-V04-3: `requestExtension()` allows extension AFTER countdown started

**File**: `FullScreenOverlayView.swift`, lines 266-274
**Severity**: MEDIUM

`requestExtension()` doesn't check `isCountingDown`. Since the button is only shown when `!isCountingDown` (line 241), this is currently safe. However, the method itself has no guard — if the button visibility logic changes, extensions could be granted during an active countdown.

**Impact**: No current bug. Defensive guard recommended.

#### M-V04-4: Hardcoded health score in `showTier3Fullscreen`

**File**: `NotificationManager.swift`, line 258
**Severity**: MEDIUM

`escalateToTier3(healthScore: 65)` passes a hardcoded `65` instead of reading the actual health score from `BreakScheduler`. The Tier 3 overlay displays "Current Eye Health: 65/100" regardless of the real value.

**Impact**: Misleading UI. The user sees a fixed score instead of their actual health score.

**Suggested fix**: Pass the real health score from `BreakScheduler` through the notification pipeline.

#### L-V04-4: `startBreak()` sets `totalDuration = remainingSeconds` — includes extensions

**File**: `FullScreenOverlayView.swift`, line 263
**Severity**: LOW

If the user requested extensions before starting the break, `totalDuration` is reset to the extended `remainingSeconds`. This means the progress ring starts from 0 for the extended duration. This is arguably correct behavior (shows remaining work), but the initial `totalDuration` of 15*60 set in `@State` is overwritten.

**Strengths:**

1. **Tier 2 wiring**: `showTier2Overlay` correctly connects overlay callbacks to `acknowledgeBreak()` and the skip flow (cancel escalation, dismiss, clear callbacks, invoke onSkipped).
2. **Tier 3 wiring**: `showTier3Fullscreen` dismisses Tier 2 before showing Tier 3. Correct escalation sequence.

**Issues:**

#### M-V04-5: Tier 2 `onSkipped` closure duplicates `acknowledgeBreak` cleanup logic

**File**: `NotificationManager.swift`, lines 235-244
**Severity**: MEDIUM

The `onSkipped` closure in `showTier2Overlay` manually calls `cancelEscalation()`, `dismissAllOverlays()`, sets `isNotificationActive = false`, `clearCallbacks()`, and invokes `callback?()`. This is the same cleanup as `handleEscalationTimeout()` with `onSkippedCallback` replaced by the locally captured `callback`. This duplication means any future change to the cleanup sequence must be applied in both places.

**Suggested fix**: Extract cleanup to a shared method like `completeNotification(callback:)`.

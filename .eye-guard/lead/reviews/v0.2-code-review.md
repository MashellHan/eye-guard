## New Issues Found in v0.2

### MEDIUM

#### M-NEW-1: `tick()` spawns unstructured Task for activity polling

**File**: `BreakScheduler.swift`, lines 202-211
**Description**: Every tick (every second) spawns a new `Task` to poll `activityMonitor.isIdle`. If the async call takes longer than 1 second, multiple concurrent Tasks could overlap, potentially causing rapid idle/active transitions. The Task is fire-and-forget — no cancellation tracking.

**Impact**: Under normal conditions this is fine (the `isIdle` property access on an actor is fast). Could become a problem if the activity monitor's isolation is contended.

**Suggested fix**: Use a dedicated polling Task with its own cadence, or track the outstanding poll Task and skip the next poll if one is in-flight.

#### M-NEW-2: `triggerBreakNotification` uses `[weak self]` in closures

**File**: `BreakScheduler.swift`, lines 263-271
**Description**: The `onTaken` and `onSkipped` closures capture `[weak self]`. Since `BreakScheduler` is `@MainActor`, these closures correctly re-enter MainActor via `Task { @MainActor in }`. However, the `[weak self]` is unnecessary — the scheduler owns the notification sender, and the callbacks are cleared after use. The `self?` optional chaining means a break acknowledgment could silently fail to record if `self` was deallocated.

**Impact**: Low — BreakScheduler is long-lived (owned by `@State` in the App struct). But the pattern is inconsistent with the C1 fix rationale.

#### M-NEW-3: `ensureDataDirectories()` still silently swallows errors

**File**: `AppDelegate.swift`, line 66
**Description**: Carried over from v0.1 M4. `try?` on directory creation.

### LOW

#### L-NEW-1: `NotificationSending` protocol is `@MainActor`-isolated

**File**: `NotificationSending.swift`
**Description**: The protocol requires `@MainActor` conformance, which means mock implementations in tests must also be `@MainActor`. This works but couples the protocol to a specific isolation domain. A more flexible design would use `nonisolated` methods with `Sendable` constraints.

**Impact**: Low — current mock pattern works fine. May need revisiting if background notification delivery is ever needed.

#### L-NEW-2: DailyReportGenerator writes to `~/EyeGuard/reports/` during tests

**File**: `DailyReportGeneratorTests.swift`
**Description**: The `generator.generate()` calls in tests write actual files to disk. Tests should inject a mock file manager or use a temporary directory to avoid polluting the user's home directory.

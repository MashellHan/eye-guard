# v0.6 Implementation Notes

## Features

### 1. Enhanced HealthScoreCalculator

**File**: `EyeGuard/Sources/Reporting/HealthScoreCalculator.swift`

Extended the existing calculator with:

- **Score trend tracking** (`ScoreTrend` enum): Compares current score against the last 5 historical scores to determine direction (improving/declining/stable). Uses a ±3 point threshold to avoid noise.

- **Per-component breakdown** (`HealthScoreBreakdown` struct): Returns all 4 components with name, score, maxScore, and human-readable explanation text.

- **New types added**:
  - `ScoreTrend` — enum with `improving`, `stable`, `declining` cases plus `symbol` and `displayName` properties
  - `ScoreComponent` — single component with name, score, maxScore, explanation
  - `HealthScoreBreakdown` — full breakdown with score, components array, trend, summaryText

- **Explanation text generation**: Each component now generates contextual explanation text:
  - Breaks: "3/5 breaks taken (60%)"
  - Discipline: "Longest session: 45min — try shorter stretches"
  - Time: "Screen time: 6h 30m — approaching 8h limit"
  - Quality: "Average break quality: 85%"

### 2. Real-Time Score Updates

**File**: `EyeGuard/Sources/Scheduling/BreakScheduler.swift`

New properties:
- `currentTrend: ScoreTrend` — current trend direction
- `currentBreakdown: HealthScoreBreakdown?` — full breakdown with components
- `scoreHistory: [Int]` — rolling history of last 20 scores

Behavior:
- Health score recalculates every 60 seconds (tick-based counter)
- Score history records each recalculation for trend analysis
- `resetDaily()` clears all new fields

### 5. Data Persistence

**File**: `EyeGuard/Sources/Persistence/DataPersistenceManager.swift` (new)

- **DailyData struct**: Codable container for daily state — break events, screen time, longest session, score history
- **DataPersistenceManager**: Sendable struct with static factory methods for encoder/decoder (avoids Swift 6 Sendable issues)
- **File format**: `~/EyeGuard/data/YYYY-MM-DD.json` with pretty-printed, sorted-keys JSON
- **ISO 8601 dates**: BreakEvent timestamps use ISO 8601 encoding for reliable round-trip

**Integration with BreakScheduler**:

- Restores: break events, counters, screen time, longest session, score history
- Recalculates health score and breakdown from loaded data

## Architecture Decisions

1. **Static factory methods for encoder/decoder**: Swift 6 strict concurrency requires `Sendable` conformance. `FileManager`, `JSONEncoder`, `JSONDecoder` are not `Sendable`. Using static factory methods that create fresh instances avoids storing non-Sendable properties.

2. **Tick-based periodic operations**: Rather than separate timers, health score recalculation (every 60s) and data persistence (every 5min) use tick counters in the existing 1-second timer loop. This keeps the timer architecture simple and avoids timer proliferation.

3. **Score history capped at 20**: Rolling window of last 20 scores prevents unbounded growth while providing enough data for meaningful trend analysis.

4. **Trend threshold of ±3 points**: Prevents trend flickering from minor score fluctuations. Only consistent directional changes register as improving/declining.

5. **Optional healthScore parameter in overlay**: `showBreakOverlay(breakType:healthScore:)` defaults to 100 for backward compatibility with existing test code.

# v0.4 Implementation Notes: Floating Overlay Break Reminder Window (Tier 2)

## Summary

Implemented the Tier 2 notification escalation as a floating overlay window that appears when the user ignores a Tier 1 macOS notification center alert.

## Files Created

### `EyeGuard/Sources/Notifications/OverlayWindow.swift`

- `OverlayWindowController` — manages the lifecycle of a floating `NSWindow`
- Window properties:
  - `NSWindow.Level.floating` (above other windows)
  - Borderless style (`[.borderless]`)
  - `isMovableByWindowBackground = true` (drag to reposition)
  - `hasShadow = true`
  - `collectionBehavior = [.canJoinAllSpaces, .stationary]` (visible on all desktops)
  - `isReleasedWhenClosed = false` (controller manages lifecycle)
  - Clear/transparent background
- Position: top-right corner of main screen with 20pt margin
- Smooth fade-in (0.3s ease-out) on show, fade-out (0.3s ease-in) on dismiss
- Replaces existing overlay if `showBreakOverlay` is called while one is active

### `EyeGuard/Sources/Notifications/BreakOverlayView.swift`

- `BreakOverlayView` — SwiftUI view hosted inside the overlay window
- Content:
  - Break-type-specific SF Symbol icon (eye, cup.and.saucer, figure.walk)
  - Title: "Time for an eye break!"
  - Contextual subtitle based on break type
  - Countdown timer with numeric text content transition
  - Progress bar with color transitions (orange → blue → green)
  - "Take Break" (borderedProminent) and "Skip" (bordered) buttons
- Visual effects:
  - `.ultraThinMaterial` background (semi-transparent blur)
  - Rounded rectangle clip shape (16pt corner radius)
  - Subtle white border stroke (0.2 opacity)
  - Drop shadow
  - Spring scale-in animation on appear
  - Pulsing icon animation during active break
- Behavior:
  - "Take Break" starts countdown from break duration
  - Timer ticks every 1 second
  - On completion: calls `onTaken`, dismisses overlay
  - "Skip" button: calls `onSkipped`, dismisses immediately
  - Timer invalidated on view disappear (cleanup)

## Architecture Decisions

1. **NSWindow + NSHostingView pattern**: Used NSWindow hosting SwiftUI view rather than pure SwiftUI WindowGroup because:
   - Menu-bar-only apps (LSUIElement) don't have standard window management
   - Need precise control over window level, collection behavior, borderless style
   - SwiftUI `Window` scene type doesn't support floating/always-on-top

2. **@MainActor throughout**: Both `OverlayWindowController` and `NotificationManager` are `@MainActor` since all UI and AppKit operations must run on main thread. Swift 6 strict concurrency enforced.

3. **Callback wiring**: The overlay's Skip and Take Break buttons connect back through the `NotificationManager`'s existing callback system (`onTakenCallback` / `onSkippedCallback`) to properly trigger `BreakScheduler` state updates.

4. **Timer-based countdown**: Used `Timer.scheduledTimer` inside SwiftUI view for the countdown rather than `Task.sleep` to keep the view self-contained and avoid lifecycle issues.

## Escalation Flow

```
Break Due → Tier 1 (macOS notification)
         → wait 2 min
         → Tier 2 (floating overlay with countdown)
         → wait 5 min
         → Tier 3 (full-screen, mandatory only) [TODO]
         → timeout → onSkipped()
```
